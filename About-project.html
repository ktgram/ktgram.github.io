The principles that guided the creation.

Less time to the bot interface - More time to the business logic behind it.

No matter which method of processing you prefer manually or through annotations, the process from idea to implementation is as short as possible.

The interface should be lightweight.

The library tries to be flexible and provide the ability to do what you need, but the choice of what you need and the ability to implement it is left to the user.

For example, exception handling or update filtering.

Approach of library interface building

Mirroring.

The approach used to create the methods of the library interface implies the maximum mirroring to the Telegram Bot Api, which allows you to not get hung up on understanding the methods of the library itself, but simply use the api as is.

The only thing is that for methods that have send* at the beginning are implemented as shortcuts, more often methods without that keyword have interface is more convenient. I.e. it is preferable to use message instead of sendMessage etc.

Data consistency.

All of the optional parameters are placed in additional sections and/or marked as nullable accordingly.

Asynchronous

Regardless of which sendAsync/send methods you use, all operations are suspendable and asynchronous, and the library implies the use of nonblocking i/o.

As you may have noticed the difference between the two sending methods is that with sendAsync it is possible to handle Response.

Commands

In fact, the concept of commands in the interface of the library is slightly different from that in the Telegram Bot Api, it is more extensive and involves checking for keywords not only text but also callback queries.

Because in essence callback queries act as the same entity.

But if you want you can use the CallbackQuery nested annotation (@CommandHandler.CallbackQuery or import and use as is), it's kind of a shortcut with a preset scope.
